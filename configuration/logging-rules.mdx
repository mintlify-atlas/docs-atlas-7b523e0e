---
title: "Logging Rules"
description: "Control which log events go to which targets using NLog logging rules."
---

A logging rule (`<logger>` in XML, `LoggingRule` in code) is the core routing mechanism in NLog. Each rule specifies which loggers it applies to, which severity levels it accepts, and which targets receive the matched events.

## Rule structure

A rule has three key parts:

1. **Logger name pattern** — which loggers the rule matches
2. **Level filter** — which severity levels pass through
3. **Target list** — where matched events are sent

```xml
<logger name="MyApp.*" minlevel="Info" writeTo="file,console" />
```

Rules are evaluated **in order**, from top to bottom in the `<rules>` section. Every rule whose name pattern and level filter match the event is executed — unless a `final` rule is encountered first.

## Logger name matching

The `name` attribute accepts exact names, wildcards (`*` for any sequence of characters, `?` for exactly one character), or prefix patterns.

| Pattern | Matches |
|---|---|
| `*` | Every logger |
| `MyApp.*` | All loggers whose name starts with `MyApp.` |
| `MyApp.Services.OrderService` | Exactly that one logger |
| `*.Repository` | Any logger whose name ends with `.Repository` |
| `MyApp.?.Controller` | Single-character wildcard in the middle |

```xml
<!-- Catch all loggers under the MyApp namespace -->
<logger name="MyApp.*" minlevel="Debug" writeTo="appFile" />

<!-- Silence all Microsoft framework internals -->
<logger name="Microsoft.*" maxlevel="Info" writeTo="blackhole" final="true" />
```

In code, the pattern is set via `LoggingRule.LoggerNamePattern` and supports the same `*` and `?` wildcards:

```csharp
var rule = new LoggingRule("MyApp.*", LogLevel.Debug, LogLevel.Fatal, target);
```

## Level attributes

### minlevel / maxlevel

Include events within an inclusive range:

```xml
<!-- Info, Warn, Error, Fatal -->
<logger name="*" minlevel="Info" writeTo="console" />

<!-- Warn and Error only -->
<logger name="*" minlevel="Warn" maxlevel="Error" writeTo="alertLog" />
```

### level

Match a single exact level:

```xml
<logger name="*" level="Fatal" writeTo="alertTarget" />
```

### levels

Match a comma-separated list of specific levels:

```xml
<logger name="*" levels="Trace,Fatal" writeTo="extremes" />
```

## The `final` attribute

When `final="true"`, no further rules are evaluated for log events that match this rule. This is the primary way to suppress noisy namespaces:

```xml
<!-- Silence verbose Microsoft logs, then stop. Nothing below sees them. -->
<logger name="Microsoft.*" maxlevel="Info" writeTo="blackhole" final="true" />

<!-- This rule never runs for Microsoft.* loggers -->
<logger name="*" minlevel="Trace" writeTo="everything" />
```

<Note>
  `final` affects rule evaluation for the matching event, not the entire logger. A `Fatal` message from `Microsoft.*` would still be evaluated by rules below if `maxlevel="Info"` means the first rule does not match at `Fatal` severity.
</Note>

## The `enabled` attribute

Disable a rule without removing it from the config:

```xml
<logger name="*" minlevel="Trace" writeTo="verboseFile" enabled="false" />
```

Useful for toggling debug rules in config without a code change.

## The `writeTo` attribute

A comma-separated list of target names declared in `<targets>`:

```xml
<logger name="*" minlevel="Info" writeTo="console,file,database" />
```

In code, use `LoggingRule.Targets` or the `AddRule` / `WriteTo` APIs:

```csharp
// Single target
config.AddRule(LogLevel.Info, LogLevel.Fatal, consoleTarget);

// Multiple targets on one rule
var rule = new LoggingRule("*", LogLevel.Info, LogLevel.Fatal, consoleTarget);
rule.Targets.Add(fileTarget);
config.AddRule(rule);
```

## Filters within a rule

Rules can carry fine-grained `<filters>` that further control whether a matched event is written. Filters are evaluated after the level and name checks pass.

```xml
<logger name="*" minlevel="Debug" writeTo="console">
  <filters defaultAction="Log">
    <when condition="contains('${message}','password')" action="Ignore" />
    <when condition="level == LogLevel.Debug and starts-with('${logger}','MyApp.Data')" action="Ignore" />
  </filters>
</logger>
```

### Filter types (XML)

| Element | Description |
|---|---|
| `<when condition="..." action="..." />` | Evaluate an NLog condition expression |
| `<whenContains layout="..." substring="..." action="..." />` | Match when the rendered layout contains a substring |
| `<whenEqual layout="..." compareTo="..." action="..." />` | Match when the rendered layout equals a value |
| `<whenNotContains layout="..." substring="..." action="..." />` | Match when the rendered layout does not contain a substring |
| `<whenNotEqual layout="..." compareTo="..." action="..." />` | Match when the rendered layout does not equal a value |
| `<whenRepeated layout="..." action="..." />` | Suppress duplicate messages within a time window |

### FilterResult values

Each filter specifies an `action` that determines what happens when the filter matches:

| FilterResult | Behaviour |
|---|---|
| `Log` | Write the event. Stop evaluating further filters. |
| `Ignore` | Drop the event for this rule only. Continue with next rule. |
| `LogFinal` | Write the event. Also stop evaluating any further rules (`final` behaviour). |
| `IgnoreFinal` | Drop the event. Also stop evaluating any further rules. |
| `Neutral` | No decision — continue evaluating the next filter. |

### defaultAction

`defaultAction` (or `filterDefaultAction` in the `<logger>` element) controls what happens when no filter matches:

```xml
<logger name="*" minlevel="Debug" writeTo="console" filterDefaultAction="Log">
  <filters>
    <when condition="contains('${message}','sensitive')" action="Ignore" />
  </filters>
</logger>
```

<Note>
  As of NLog 5, the default value of `filterDefaultAction` changed from `Neutral` to `Ignore`. If you define filters but do not specify `filterDefaultAction`, unmatched events are dropped for that rule. Set `filterDefaultAction="Log"` explicitly if you want unmatched events to be written.
</Note>

### Programmatic filters

Use `FilterDynamic` in the fluent API for delegate-based filtering:

```csharp
LogManager.Setup().LoadConfiguration(c => c
    .ForLogger()
        .FilterMinLevel(LogLevel.Debug)
        .FilterDynamicIgnore(evt => evt.Message.Contains("password"))
        .WriteToConsole());
```

## Rule ordering and evaluation

<Steps>
  <Step title="Level gate (GlobalThreshold)">
    Before any rules run, `LogManager.GlobalThreshold` is checked. Events below the global threshold are discarded.
  </Step>
  <Step title="Iterate rules top to bottom">
    Each rule is checked. If the logger name matches and the level is enabled for the rule, the rule fires.
  </Step>
  <Step title="Apply filters">
    If the rule has `<filters>`, they are evaluated in order. The first filter to return a non-`Neutral` result determines whether the event is written.
  </Step>
  <Step title="Write to targets">
    Events that pass level and filter checks are written to all targets listed in `writeTo`.
  </Step>
  <Step title="Stop on final">
    If the matching rule has `final="true"`, evaluation stops. No further rules are checked for this event.
  </Step>
</Steps>

## Common rule patterns

<CodeGroup>
```xml Silence third-party noise
<rules>
  <logger name="Microsoft.*"    maxlevel="Info" writeTo="blackhole" final="true" />
  <logger name="System.Net.Http.*" maxlevel="Info" writeTo="blackhole" final="true" />
  <logger name="*" minlevel="Info" writeTo="console,file" />
</rules>
```

```xml Separate error log
<rules>
  <logger name="*" minlevel="Error" writeTo="errorFile" />
  <logger name="*" minlevel="Info"  writeTo="appFile,console" />
</rules>
```

```xml Single namespace at Trace, everything else at Info
<rules>
  <logger name="MyApp.Data.*" minlevel="Trace" writeTo="debugFile" final="true" />
  <logger name="*"            minlevel="Info"  writeTo="appFile" />
</rules>
```

```xml Fatal-only alert channel
<rules>
  <logger name="*" level="Fatal" writeTo="alertTarget" />
  <logger name="*" minlevel="Info" writeTo="appFile" />
</rules>
```
</CodeGroup>
